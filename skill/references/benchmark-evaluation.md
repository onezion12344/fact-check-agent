# Benchmark Evaluation Methodology Reference

> Captured from Harry's 2026-06-23 critique of single-LLM benchmark approach.
> This reference documents the industry standard methodologies and the correct approach for evaluating the Double-Check pipeline.

## The Problem Harry Identified

原始 benchmark（v2 + v3）的方法论是：我用 DeepSeek V4 Flash 回答 40 题 → 我用 web_search 验证 → **我（同一 LLM）判分**。Harry 指出这是循环论证：

> "关键是我们涉及到的事实很多样，逻辑也很多样，没有办法用简单程序化的方式去查，还是得用人工智能去判断人工智能。"

他的结论对：事实多样性（价格、规格、统计数据、政治数据、逻辑谬误）决定了无法用脚本化判分。但用同一个 LLM 既生产答案又判断对错，是循环。

## Industry Approaches

### 1. SimpleQA (OpenAI, Oct 2024)

**Source:** cdn.openai.com/papers/simpleqa.pdf / arxiv.org/abs/2411.04368

| Aspect | Detail |
|:-------|:-------|
| Scale | 4,326 questions |
| Domains | History, science, technology, art, entertainment |
| Ground truth | Human-annotated — each question has exactly one correct answer |
| Judge | **No LLM involved.** String matching against human ground truth |
| Metric | accuracy = correct / (correct + incorrect + not attempted) |
| Limitation | Only works for "answers that don't change over time" — excludes prices, hours, current events |

SimpleQA avoids the LLM-judge-LLM problem by constraining the task: every question has a single, immutable, human-verified answer. This is the gold standard for **closed-domain** factuality but is useless for Double-Check's domain (live data, prices, subjective statistics).

### 2. FACTS Grounding (Google DeepMind, Dec 2024)

**Source:** deepmind.google/blog/facts-grounding / arxiv.org/abs/2501.03200

| Aspect | Detail |
|:-------|:-------|
| Scale | 1,719 examples (860 public, 859 private) |
| Format | Document + system instruction + user request |
| Max context | 32,000 tokens |
| Domains | Finance, technology, retail, medicine, law |
| Ground truth | Human-annotated reference segments |
| Judge | **3 frontier LLMs: Gemini 1.5 Pro, GPT-4o, Claude 3.5 Sonnet** |
| Calibration | Judge models evaluated against held-out test set; prompts verified against human raters |
| Two-phase | (1) Eligibility check — does the response address the user request? (2) Grounding check — is it factually accurate? |
| Score | Average of all 3 judge models across all examples |

FACTS explicitly acknowledges LLM judge bias: *"We selected a combination of different judges to mitigate any potential bias of a judge giving higher scores to the responses produced by a member of its own model family."* This is the closest methodological match for Double-Check.

### 3. HHEM (Vectara, 2023-2026)

**Source:** vectara.com/blog/hhem-2-1 / huggingface.co/vectara/hallucination_evaluation_model

| Aspect | Detail |
|:-------|:-------|
| Architecture | DeBERTaV3 (fine-tuned) |
| Training data | Synthetic data generated by FLAN-T5 XXL |
| Task | Binary classification: is this summary factually consistent with the source? |
| Output | Score 0.0-1.0 |
| Versions | HHEM-2.1-Open (open source), HHEM-2.3 (commercial) |
| Not a general LLM | It's a specialized classifier, not a chat model |

HHEM solves the LLM-judge-LLM problem by being purpose-built for ONE task. It's not "a model asked to judge" — it's a model trained specifically for factual consistency detection.

## Recommended Approach for Double-Check

### 4-Layer Methodology

```
Layer 1 — Human Calibration Set
  Harry (or another human) creates ground truth for N=20-40 questions
  Each: question + correct answer + answer type + source URL
  Covers all 4 categories: prices, specs, stats, schedules
  = the absolute reference frame

Layer 2 — Multi-Judge Panel
  At minimum 3 judges from different model families:
    Judge A: DeepSeek V4 Flash (current model)
    Judge B: DeepSeek V4 Pro (upgrade)
    Judge C: MiMo V2.5 Pro or Claude (different family)
  Each independently evaluates raw output AND pipeline output
  Record per-question verdicts in a matrix

Layer 3 — Disagreement Arbitration
  All 3 agree → accepted
  2:1 split → flagged "disputed", human arbitrates
  All 3 disagree → question design problem, remove or rephrase

Layer 4 — Transparent Report
  Quantitative: Δ accuracy, Δ fallacy rate, tool call cost
  Qualitative: disagreement analysis, edge case catalog
  Statement: "LLM judges have known biases. This report constrains bias via multi-judge panel + human calibration + transparent disagreement reporting."
```

### What NOT to Do (Deprecated)

❌ Single LLM answers → same LLM verifies → same LLM scores
→ Circular. This is what the v2/v3 benchmarks did. Harry identified and rejected it.

❌ Claiming 100% accuracy from a pipeline evaluated by the pipeline's own model
→ Self-serving. Always use independent judges for the final score.

### Reporting Template

```markdown
# Benchmark Report

## Model: [name]
## Judge Panel: [A, B, C]
## Calibration: [N] human-annotated questions

| Metric | Raw | + Pipeline | Δ |
|:-------|:---:|:----------:|:-:|
| Factual accuracy | XX% | XX% | +XXpp |
| Logical consistency | XX% | XX% | +XXpp |
| Tool calls per query | 0 | XX | — |

## Disagreement Analysis
- Questions with unanimous agreement: X/N
- Questions with 2:1 split: X/N (arbitrated by human)
- Questions with full disagreement: X/N (removed)

## Limitations
- LLM judges have known biases toward their own model family
- Human calibration set is small (N=XX)
- Time-sensitive facts may change between benchmark run and reading
```
